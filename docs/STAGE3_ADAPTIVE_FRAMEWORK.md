# Stage 3: Controlled Self-Evaluating, Self-Improving Framework

Stage 3 upgrades the bot from a tradeable live system into a **controlled self-evaluating, self-improving architecture**. It does **not** implement unconstrained AI trading or reckless self-modification.

## What Stage 3 Does

- **Config versioning**: Every config is versioned, fingerprinted, and traceable. Live runs are associated with a config version.
- **Evaluation**: Compute core and stratified metrics from stored trades/fills/decisions; write JSON/CSV/MD reports and DB records.
- **Dataset / replay**: Build analysis datasets from stored data; approximate replay of decision logic (not tick-accurate).
- **Walk-forward optimization**: Bounded parameter search (grid/random) over approved params only; train/validation/test splits.
- **Overfitting guardrails**: Reject or penalize candidates with too few trades, OOS degradation, symbol concentration, etc.
- **Candidate config generation**: Generate candidates from a parent config; only approved fields may change; artifacts on disk.
- **Shadow comparison**: Run a candidate config alongside the baseline (same inputs, no live orders); persist and compare decisions. **Implemented as post-hoc**: shadow run is created and closed via CLI; decisions are recorded when the live bot (or an external process) writes them; comparison report uses stored baseline vs shadow decisions. There is no in-process live parallel scoring in the current implementation.
- **Promotion workflow**: Candidate → shadow → evaluation → staged → manual (or strict auto) promotion to active; previous kept for rollback.
- **Live degradation monitoring**: Compare active config to thresholds every 5 minutes using the last 24h of stored trades; persist degradation events. **Requires a minimum number of trades** (default 5) in the window before any check is performed; otherwise status is "insufficient data" and no event is persisted.
- **Rollback protection**: One-command rollback to previous active; events persisted; no silent config flip.

## What Stage 3 Does NOT Do

- It does **not** let the strategy mutate itself arbitrarily.
- It does **not** auto-promote from tiny samples (default is manual approval).
- It does **not** change active config without versioning and persistence.
- Replay is **not** tick-accurate; it uses stored observations.

## Config Lifecycle

- **baseline** – Initial or reference config.
- **candidate** – Generated by optimizer or manually; not yet active.
- **staged** – Passed evaluation; ready for promotion.
- **active** – Currently used for live trading (only one at a time).
- **rejected** – Failed guardrails or manual reject.
- **rolled_back** – Was active, then rolled back.
- **archived** – Superseded by another active.

## Safe Operator Workflow (Ubuntu CLI)

1. **Bootstrap**: First run registers the loaded config as active if none exists.
2. **Evaluate**: `python3 run_bot.py evaluate --from-date 2025-01-01 --to-date 2025-01-31`
3. **Optimize**: `python3 run_bot.py optimize run --config-id <baseline> --from-date ... --to-date ...`
4. **Shadow** (optional): **Post-hoc mode.** Create a shadow run: `python3 run_bot.py shadow start <candidate_config_id>` (or `shadow start --candidate-config-id <id>`). Stop it when done: `python3 run_bot.py shadow stop --candidate-config-id <id>`. Report: `python3 run_bot.py shadow report <candidate_config_id>`.
5. **Promote**: After review, `python3 run_bot.py promote --config-id <candidate_config_id>`. Show active config: `python3 run_bot.py promote status`.
6. **Rollback** if needed: `python3 run_bot.py rollback`

All major decisions are persisted in SQLite and artifacts under `artifacts/`.
