# Known Limitations and Next Improvements

## Current Limitations

1. **Kline format**: Bybit returns list of lists for klines; instrument refresh may re-fetch full universe and change symbol set during run
2. **Correlation filter**: max_correlation_positions not implemented; Stage 4 adds **cluster control** (max positions per cluster via correlation proxy), not full correlation matrix
3. **Backtest**: Uses saved trades only; no full replay from trade stream
4. **Stage 3 replay**: Replay uses stored observations; not tick-accurate; no full market re-simulation
5. **Shadow**: Shadow runner records decisions in DB; comparison is **post-hoc** from stored data (no in-process live parallel candidate scoring). Use `shadow start` / `shadow stop --candidate-config-id` / `shadow report` to create runs and generate comparison reports.
6. **Degradation monitor**: Requires a minimum number of trades (default 5) in the evaluation window; otherwise no check is performed and status is "insufficient data".
7. **Stage 4**: Regime and clustering use single-snapshot or short-window proxies; evaluation by exit reason uses event counts (no direct PnL per exit reason without schema change). Optimizer samples params over same stored data unless backtest-per-candidate is added.
8. **Stage 5**: Allocator operates on a **candidate set** (multiple candidates per cycle; higher score preferred when budgets are tight). Exposure controls are lightweight. Heartbeat is **written by runtime loops** (context refresh, WS, reconciliation, lifecycle, score/entry, degradation monitor). Replay/backtest uses stored trades + fill model (not tick-accurate). Evaluation includes resized_by_allocation and allocation_method_usage counts.
9. **Burn-in**: Execution audit and protection audit are best-effort; readiness classification is conservative and heuristic. No automatic escalation of phase; operator must adjust config. The repo uses two **operating modes** — **demo_research** (autonomous Demo research/tuning) and **live_guarded** (guarded live; stricter effective profile). The **promote-env** helper (Demo -> Live) does **not** auto-promote: it requires explicit `--confirm-live` and checks readiness (READY_FOR_SMALL_LIVE) and live credentials first. **Execution mode**: `dry_run` is the single control for simulated vs real orders; `mode=paper` with `dry_run: false` places real Demo orders when `BYBIT_ENV=demo`.
10. **Credentials**: **Dual-key** (BYBIT_DEMO_API_KEY/SECRET + BYBIT_LIVE_API_KEY/SECRET) is recommended; legacy single BYBIT_API_KEY/SECRET or BYBIT_ENV=testnet with testnet keys is supported as fallback. The system does not auto-switch from demo to live; operator must set BYBIT_ENV=live and have live keys before guarded live start.
11. **Demo automation**: The optional Demo orchestration (`automation.enabled`, `automation.demo_orchestration_enabled`) only runs readiness/evaluation/optimizer/shadow and writes recommendation artifacts under `artifacts/automation/` (or `artifacts/demo/automation/` when using the Demo instance). It never auto-promotes configs or environments **to Live**; the operator must still run **`python run_bot.py promote-to-live --candidate-config-id <id> ... [--activate]`** to import a Demo candidate into Live (and optionally activate it), and `python run_bot.py promote-env --confirm-live` for environment promotion. **Demo auto-adopt** (when `automation.auto_adopt_demo_candidates` is true) allows the Demo instance to automatically activate a better candidate as its **own** active config (Demo DB only); adoption is controlled by `min_trades_for_demo_adoption`, `min_hours_between_demo_adoptions`, and optionally `require_shadow_before_demo_adoption`. The new config is used **after the next Demo restart**. Live is never modified by Demo auto-adopt. **Fixed-equity Demo research:** When `operating_mode: demo_research` and `demo_research.fixed_equity_enabled` is true, the Demo instance uses a fixed synthetic equity (e.g. $1000) for **sizing and risk budgeting** instead of the wallet balance; kill-switch and drawdown checks still use the actual account. `demo_research.relaxed_kill_switch_enabled` applies Demo-only wider kill-switch thresholds; `demo_research.demo_research_burnin_permissive` raises burn-in limits for long-running research. Live is never affected. A **systemd timer** (`money-flow-momentum-automation.timer` or `money-flow-momentum-demo-automation.timer` for dual-instance) can run `python run_bot.py automation cycle` periodically (e.g. every 15 min); the timer does not start the trading bot and does not auto-promote. Trading and orchestration are separate services. **Dual-instance:** Demo and Live can run simultaneously on one host with isolated DB, artifacts, logs, and heartbeat; only the Demo instance runs the automation timer.
12. **Warm-start**: **Demo-only** historical-candle calibration before first Demo trading. When enabled (`warm_start.enabled`) and local Demo trade count is below `warm_start.min_local_trades_to_skip_warm_start`, the system fetches (or loads from cache) historical candles, runs a calibration/optimization pass on synthetic trades derived from candles, and activates a seed config for Demo. If no viable seed is found and `warm_start.fallback_to_safe_seed_on_failure` is true, a conservative fallback config is activated. **Live is never touched**; no auto-promotion to Live. Warm-start requires Demo API credentials to fetch candles (or pre-cached local data). Report: `artifacts/<instance>/warm_start/warm_start_report.json`.

## Next Improvements

1. **Correlation**: Stage 4 cluster control uses correlation proxy; optional full rolling correlation matrix
2. **Backtest**: Replay from saved trade stream or kline aggregates
3. **PostgreSQL**: Optional storage backend
4. **Prometheus**: Metrics for observability
